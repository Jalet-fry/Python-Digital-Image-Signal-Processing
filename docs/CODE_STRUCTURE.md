# Разбор архитектуры кода PythonDSP

Этот документ объясняет, как устроены внутренности проекта, без лишних комментариев в коде.

## 1. Папка `core/signals/` (Сердце проекта)
Здесь лежат математические движки. Они не знают о графиках, они только считают числа.

### `generator.py`
* **Что делает:** Создает массив чисел (сигнал).
* **Логика:** В цикле `for` считается формула синуса для каждой гармоники и всё складывается в один массив.

### `fourier.py`
* **`dft(x)`:** Использует матричное умножение (`np.dot`). Это "честный" способ по формуле из учебника.
* **`fft(x)`:** Рекурсивная функция. Она вызывает саму себя, деля массив пополам, пока не дойдет до 1 элемента. Это и есть "разделяй и властвуй".
* **`ifft(x)`:** Хитрость. Вместо написания новой функции, мы просто берем сопряженное число, прогоняем через обычное БПФ и снова берем сопряженное.

### `math_ops.py`
* **`linear_convolution`:** Два вложенных цикла `for`. Внутренний цикл перемножает точки, внешний — сдвигает сигнал.
* **`fft_convolution`:** Использует "Теорему о свертке". Просто перемножает результаты БПФ.

## 2. Папка `labs/` (Интерфейс и Логика лабы)

### `lab1_instruments.py`
Это диспетчер. Его работа:
1. Вызвать математику из `core`.
2. Сгруппировать результаты в 8 блоков (по 3 графика).
3. Почистить фазу от мусора (порог 0.1%).
4. Сохранить 24 картинки в `results/plots/`.
5. Записать ваши успехи в `lab1_report.txt`.

## 3. Почему код написан именно так?
* **Чистота:** Математика отдельно, рисование отдельно.
* **Точность:** Мы используем `float64`, чтобы ошибка не превышала $10^{-13}$ (эпсилон).
* **Наглядность:** Группировка по 3 графика позволяет мгновенно увидеть совпадение теоремы с практикой.
